# Branch Review Guide

이 문서는 이 저장소에서 GPT 기반 에이전트가 브랜치 작업 내용을 리뷰할 때 따를 기준을 정의합니다.

## 목적

브랜치 작업 결과를 머지하거나 푸시하거나 GitHub에 올리기 전에 리뷰할 때 이 문서를 사용합니다.

리뷰의 핵심은 해당 브랜치가 안전한지, 정상 동작할 가능성이 높은지, 공개해도 되는 상태인지 확인하는 것입니다.

## 브랜치 리뷰 목표

모든 브랜치 리뷰에서 다음 항목을 확인합니다.

1. 중요한 보안 문제가 없는지
2. 현재 프로젝트 구조에서 코드가 원활하게 동작할 가능성이 높은지
3. GitHub에 올리면 안 되는 민감 정보나 프라이빗한 내용이 없는지
4. Docker 관련 파일과 의존성 파일이 여전히 안전하고 타당한지
5. 환경변수 사용 방식이 코드와 배포 가정에 맞는지

## 필수 리뷰 영역

### 1. 보안 리뷰

다음 항목을 확인합니다.

- 하드코딩된 비밀값, 토큰, API 키, 비밀번호, 비공개 URL
- JWT 비밀값 노출이나 인증 우회 가능성
- 안전하지 않은 파일 접근, 셸 실행, 경로 처리
- 검증되지 않은 외부 입력이 모델/OCR/DB/서브프로세스 로직까지 직접 들어가는지
- 민감한 요청 데이터나 사용자 데이터가 로그에 남는지
- 서버 주소, SSH 관련 정보, 내부 전용 엔드포인트, 인증 정보 포함 설정이 실수로 노출되는지

이 저장소에서 특히 중점적으로 볼 파일:

- `core/auth.py`
- `core/settings.py`
- `.env` 사용 방식
- JWT, DB 설정, Ollama 주소, 파일 업로드 처리, OCR 입력 경로를 다루는 코드 전반

### 2. 동작 / 기능 리뷰

다음 항목을 확인합니다.

- 깨진 import, 이름이 바뀐 모듈, 잘못 연결된 함수 호출
- `main.py`와 `app_oss_main.py` 사이의 두 서비스 분리가 깨지는 변경
- 챗봇 응답 포맷 회귀
- RAG 동작을 깨뜨릴 수 있는 검색/인덱싱 변경
- 시간표 추출을 깨뜨릴 수 있는 OCR 파이프라인 변경
- 규칙 기반/ML 기반 텍스트 필터링 불일치

이 저장소에서 특히 중점적으로 볼 파일:

- `main.py`
- `app_oss_main.py`
- `LLM/OSS/Open_AI_OSS.py`
- `LLM/OSS/formatter.py`
- `LLM/OSS/postprocess/`
- `LLM/sub_model/`
- `image_analysis/`
- `text_filtering/`

### 3. GitHub 공개 / 프라이버시 리뷰

브랜치를 안전하게 업로드해도 되는지 판단하기 전에 다음 항목을 확인합니다.

- 소스 코드, 주석, 예제, 노트북, 테스트 데이터 안의 비밀정보
- 내부 호스트명, 사설 IP, 운영 서버 정보, SSH 접속 정보, 비공개 문서 내용
- 로컬 절대 경로, 사용자명, 특정 머신 전용 설정, 조직 내부 메모
- 외부 공개되면 안 되는 생성 데이터, 모델 아티팩트, 원본 데이터셋, 백업 파일
- `.env`, 키 파일, 토큰, 디버그 덤프, 내부 로그가 실수로 포함되었는지

이 저장소에서 특히 주의할 영역:

- `data/`, `model/`, `database/`, `debug/`, `backup_file/`, `.env`
- 서버 접속 예시나 인증정보처럼 보이는 문자열이 추적 파일에 새로 들어가지 않는지
- `.gitignore`가 프라이빗 파일과 생성 산출물을 계속 적절히 보호하는지

### 4. Dockerfile 및 의존성 리뷰

항상 다음 파일을 확인합니다.

- `Dockerfile`
- `Dockerfiles_oss`
- `requirements.txt`
- `requirements_oss.txt`

다음 항목을 확인합니다.

- 불필요한 패키지 추가
- 의존성 버전 충돌
- 로컬 환경과 컨테이너 환경 사이의 실행 가정 불일치
- OCR, Java, 모델/런타임에 필요한 시스템 패키지 누락
- 수상한 설치 소스나 지나치게 느슨한 버전 고정
- 빌드 중 비밀값이 노출될 수 있는 명령
- 이미지 비대화나 피할 수 있는 취약한 기본 설정

### 5. 환경변수 리뷰

다음 항목을 확인합니다.

- 새로 추가된 환경변수가 문서화되지 않았거나 settings 코드에서 읽히지 않는지
- 제거되거나 이름이 바뀐 환경변수가 코드에서 여전히 참조되는지
- `.env`, `core/settings.py`, 런타임 코드, Docker 설정이 서로 맞지 않는지
- 기본값 누락, 잘못된 가정, 시작 자체를 깨뜨리는 필수값 문제
- 원래 환경변수로 관리해야 할 비밀값이 소스코드로 옮겨지지 않았는지
- 경로 기반 환경변수가 현재 아티팩트/데이터 로딩 로직과 어긋나지 않는지

이 저장소에서 특히 중점적으로 볼 파일:

- `core/settings.py`
- `.env`
- `main.py`
- `app_oss_main.py`
- `OSS_BASE_URL`, `OSS_MODEL`, JWT 설정, DB 설정, 아티팩트 경로, 학사일정/데이터 경로를 사용하는 코드 전반

## 권장 브랜치 리뷰 절차

### 1단계. 변경 범위 파악

다음 항목을 먼저 봅니다.

- 변경된 파일
- 영향 받는 서비스 경계
- 설정 및 의존성 변경
- 환경변수 추가, 제거, 이름 변경

이 단계에서 답해야 할 질문:

- 이 브랜치가 서비스 A, 서비스 B, 또는 둘 다에 영향을 주는가
- 런타임 동작 변경인가, 단순 포맷 변경인가, 인프라 변경인가
- 보안 민감 파일을 건드렸는가
- `.env` 업데이트나 배포 설정 변경이 필요한가

### 2단계. 보안을 가장 먼저 리뷰

스타일 의견이나 리팩토링 의견보다 보안 이슈를 먼저 봅니다.

비밀정보 유출 가능성이나 위험한 실행 경로가 조금이라도 보이면 가장 먼저 보고합니다.

### 3단계. 기능 정상 동작 여부 리뷰

변경된 요청 흐름이나 처리 경로를 따라가며, 코드가 기존 아키텍처에 맞게 동작하는지 확인합니다.

예시:

- 챗봇 응답 변경은 retrieval -> formatting -> postprocess 흐름으로 따라갑니다.
- RAG 인덱싱 변경은 `index_utils.py`의 정제/분류/청킹 -> `build_index.py`의 아티팩트 생성 -> `query_index.py`의 검색/재랭킹 -> `LLM/OSS/postprocess/` 흐름으로 따라갑니다.
- `search_df.parquet` 스키마가 바뀌면 구버전 아티팩트 fallback 여부와 `search_df`, `embeddings.npy`, `bm25.pkl` row 수 정합성을 확인합니다.
- OCR 변경은 service -> OCR engine -> timetable parsing 흐름으로 확인합니다.
- 텍스트 필터 변경은 규칙 기반과 ML 기반 양쪽을 함께 봅니다.

### 4단계. 공개 가능 상태인지 리뷰

이 브랜치가 GitHub에 올라갈 수 있다고 가정하고, 공개해도 되는 상태인지 확인합니다.

로컬에만 남아야 할 것이 있으면 명확하게 지적합니다.

RAG 품질 검증을 위한 회귀 케이스는 공개 가능한 질문/기대 문구만 포함해야 합니다. 반대로 `model/artifacts/`의 생성 산출물, 원본 데이터셋, 로컬 `.env` 값은 브랜치에 포함하지 않습니다.

### 5단계. Docker 및 requirements 리뷰

의존성이나 컨테이너가 바뀌었다면, 애플리케이션 요구사항에 여전히 맞는지와 배포 리스크가 없는지 확인합니다.

### 6단계. 환경변수 정합성 리뷰

다음 항목을 확인합니다.

- 필요한 환경변수가 실제 코드에서 사용되는지
- 코드에서 참조하는 환경변수가 정의되었거나 문서화되어 있는지
- Docker/런타임 설정이 기대한 값을 실제로 전달하는지
- 로컬 전용 값이나 비밀값이 추적 파일에 커밋되지 않았는지
- 이름이 바뀐 환경변수가 조용히 시작 실패나 런타임 오류를 만들지 않는지

## 리뷰 결과 출력 형식

리뷰 결과는 다음 우선순위로 정리합니다.

1. 보안 이슈
2. 기능 깨짐 또는 회귀 위험
3. 프라이버시 / GitHub 공개 이슈
4. Dockerfile / requirements 이슈
5. 환경변수 또는 배포 설정 불일치
6. 남아 있는 리스크 또는 미검증 항목

문제가 없다면 그 사실을 명시하고, 아직 검증하지 못한 영역이 있으면 함께 적습니다.
